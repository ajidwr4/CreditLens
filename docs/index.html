// Credit Score System

How the score
is calculated.

ON-CHAIN SCORE — MAX 700

ComponentFormulaMax
Repayment Ratio(repaidOnTime / totalLoans) × 400400
ActivitytotalLoans × 15150
VolumerepaidOnTime × 20150
Late PenaltyrepaidLate × −30—
Default Penaltydefaulted × −150—

REAL WORLD SCORE — MAX 300

ComponentFormula
On-Time RecordrealWorldOnTime × +40
Late RecordrealWorldLate × −60

DESIGN INTENT

  • Ratio ≠ Volume — Ratio rewards consistency (% on time). Volume rewards absolute activity (total count). A borrower with 5/5 and one with 50/50 score the same on ratio but differ on volume.
  • Score only recalculates on real activity — no polling, fully event-driven
  • Every update is snapshotted to scoreHistory for chart visualization
  • Real-world late penalty (−60) is higher than on-chain late (−30) — off-chain behavior carries more weight

⚠ DEFAULT PENALTY

One default = −150 points. A borrower must repay 7–8 loans on time to fully recover. Strong behavioral enforcement with no admin override.

totalScore = onChainScore + realWorldScore
Clamped to [0, 700] and [0, 300] independently

// Indexer Pipeline

From contract event
to on-chain score.

1
Contract emits event on-chain
LoanRepaid · LoanDefaulted · RecordMinted · RecordDeleted — only these 4 trigger score recalc
2
index.ts catches event ponder
Calls updateBorrowerScore(borrower, db, timestamp) — queries all loans + realWorldRecords for that address
3
scorer.ts calculates ponder
Runs formula → returns { totalScore, onChainScore, realWorldScore, tier }
4
DB upsert + snapshot ponder
creditScore table updated · scoreHistory insert (id: address-timestamp) · powers score chart
5
pusher.ts pushes on-chain on-chain
simulateContract (dry-run) → writeContract → oracle.updateScore() via dedicated wallet

WHY SIMULATE FIRST?

simulateContract runs a dry-run against the chain before broadcasting. If the transaction would revert, the error is caught before any gas is spent — the indexer keeps running without crashing.

DEDICATED PUSHER WALLET

Ponder has its own PONDER_PRIVATE_KEY — separate from any user wallet. Only this wallet can call oracle.updateScore(). If the key is not set, pusher skips silently — indexer still works, score just stays off-chain.

NON-BLOCKING DESIGN

Push failure is caught and logged — it never throws up the stack. Indexer continues processing subsequent events regardless of oracle write success.

// On-Chain Credit Gate

Score verified
trustlessly.

takeOffer() EXECUTION FLOW

📥Borrower calls takeOffer(offerId) on LendingMarket
🔍Contract calls oracle.getScore(msg.sender) — reads ScoreData struct from CreditScoreOracle
⚖️Extracts .totalScore — compares against offer.minCreditScore
❌If score < minimum → revert ScoreTooLow — transaction fails atomically
✅If score ≥ minimum → loan created, tCTC transferred to borrower in same transaction
// LendingMarket.sol — takeOffer()
ScoreData memory sd = oracle.getScore(msg.sender);
if (sd.totalScore < po.minCreditScore)
  revert ScoreTooLow(sd.totalScore, po.minCreditScore);

Why this matters

The score check happens inside the smart contract at execution time — not in the frontend, not in an off-chain API. There is no way for a borrower to bypass the gate by manipulating the UI or replaying a transaction.

The oracle stores the score pushed by Ponder. The contract reads it. No trust in any intermediary is required at any point.

FULLY ATOMIC

Score verification, loan creation, and tCTC transfer all happen in a single transaction. There is no window between check and execution — no front-running, no race condition.
